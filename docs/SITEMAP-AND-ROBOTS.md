# How the sitemap and robots.txt are created

This document describes how `public/sitemap.xml` and `public/robots.txt` are produced, where the URL list comes from, and how the project keeps them in sync with the Next.js app.

**Canonical host:** `https://www.businessvaluationexperts.co.uk` (see `lib/site.ts` ‚Üí `SITE_URL`).

## Overview

Both files are **generated by a Node script**, not maintained by hand. That keeps every indexable URL aligned with the same inventory the verification tooling uses.

| Output | Path | Generator |
|--------|------|-----------|
| XML sitemap | `public/sitemap.xml` | `scripts/generate-seo.ts` |
| robots.txt | `public/robots.txt` | `scripts/generate-seo.ts` |

Files in `public/` are served from the site root (e.g. `https://www.businessvaluationexperts.co.uk/sitemap.xml`). Next.js also exposes `/sitemap.xml` via `app/sitemap.ts`, which must list the same URLs as the committed XML file.

## URL inventory (`buildPublicUrlInventory`)

The generator imports `buildPublicUrlInventory` from `lib/seo/publicUrlInventory.ts`. That function assembles all paths that should appear in search, then maps them to absolute URLs using `SITE_URL`.

The inventory merges:

1. **Static hub pages** ‚Äî `SITEMAP_STATIC_PATHS` (home, services, case-types hub, sectors, guides, FAQ, glossary, etc.).
2. **Service detail pages** ‚Äî `/services/:slug` (8).
3. **Valuation method detail pages** ‚Äî `/valuation-methods/:slug` (3).
4. **Case type detail pages** ‚Äî `/case-types/:slug` (10 slugs from `lib/site.ts`).
5. **Sector detail pages** ‚Äî `/sectors/:slug` (8 slugs).
6. **Guide detail pages** ‚Äî `/guides/:slug` (6 slugs).

**Excluded from the sitemap** (but still routable): `/contact`, `/thank-you`, `/privacy`, `/terms`, `/cookies` ‚Äî see `NON_SITEMAP_PATHS` and `docs/SEO-ARCHITECTURE.md` ¬ß9.

The combined list is deduplicated, sorted, and exposed as:

- `sitemapPaths` / `sitemapUrls` ‚Äî indexable URLs only  
- `allPaths` / `allUrls` ‚Äî includes non-sitemap legal/contact routes for route inventory checks  

## Sitemap generation

Implementation: `lib/seo/renderSitemap.ts` (called from `scripts/generate-seo.ts`).

For each URL in `sitemapUrls`, the script emits a standard [sitemaps.org](https://www.sitemaps.org/schemas/sitemap/0.9) `<url>` entry with:

| Element | Source |
|---------|--------|
| `<loc>` | Absolute URL from inventory |
| `<lastmod>` | Editorial `updatedAt` via `getLastmodForPath()` (`lib/seo/contentDates.ts`) ‚Äî not build time |
| `<changefreq>` | `getSitemapMetaForPath()` in `lib/seo/sitemapHeuristics.ts` |
| `<priority>` | Same heuristics (e.g. home `1.0`, `/services/*` `0.85`, `/valuation-methods/*` `0.80`) |

The full document is written to `public/sitemap.xml`. The `public/` directory is created if needed.

## robots.txt generation

Implementation: `lib/seo/renderRobots.ts` (fixed template, **not** derived from the URL inventory).

| Rule | Purpose |
|------|---------|
| `User-agent: *` | All crawlers |
| `Allow: /` | Site is crawlable by default |
| `Disallow: /admin/` | No admin UI |
| `Disallow: /api/` | No API routes |
| `Disallow: /private/` | No private areas |
| `Disallow: /_next/` | Next.js build assets |
| `Disallow: /thank-you` | Post-form page (`noindex` in metadata) |
| `Sitemap: https://www.businessvaluationexperts.co.uk/sitemap.xml` | Sitemap discovery |

The file is written to `public/robots.txt`.

`renderRobotsTxt()` mirrors `app/robots.ts`: when `VERCEL_ENV=preview` or `NEXT_PUBLIC_STAGING=true`, it emits disallow-all instead of the production Allow template.

## Commands

From the repo root:

```bash
npm run seo:generate   # Regenerate public/sitemap.xml + public/robots.txt
npm run seo:verify     # Fail if committed files drift from inventory / template
npm run build          # Runs generate + verify, then next build
```

Run `seo:generate` after:

- Adding paths to `SITEMAP_STATIC_PATHS` or new slugs in `lib/site.ts`
- Changing robots disallow rules in `lib/seo/renderRobots.ts`
- Before a release when you want fresh `lastmod` dates

Commit the updated `public/sitemap.xml` and `public/robots.txt` when URLs change.

## Continuous integration

`.github/workflows/seo-checks.yml` runs on pull requests and manual dispatch:

1. `npm ci`
2. `npm run seo:generate` ‚Äî Regenerates files and catches script failures
3. `npm run seo:verify` ‚Äî Ensures `public/sitemap.xml` `<loc>` entries match `buildPublicUrlInventory().sitemapUrls` and `public/robots.txt` matches the template; also checks `app/sitemap.ts`

## Deployment (Netlify / static hosts)

`netlify.toml` sets explicit content types:

- `/sitemap.xml` ‚Üí `application/xml`
- `/robots.txt` ‚Üí `text/plain`

## Operational notes

- **Canonical host** is defined in `lib/site.ts` (`SITE_URL`) and repeated in the robots `Sitemap:` line. If the primary domain changes, update `SITE_URL` and regenerate.
- **New static pages** must be added to `SITEMAP_STATIC_PATHS` (or new slug arrays in `lib/site.ts`) or `seo:verify` will fail after the next generate.
- **Service fragment URLs** (`/services#share-equity-valuation`, etc.) are not separate sitemap entries; they are linked internally from `/services`.
- This site has **no dynamic blog/API URL sources** in the inventory (unlike larger marketing sites). All sitemap URLs come from the static lists above.
